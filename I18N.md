# Internationalization (i18n)

The site ships in **English, French, Spanish, Italian, German**. Translations are auto-generated at build time from a single English source and served as static JSON — no runtime translation cost, full SSG, full SEO.

- English at `/` (default locale, no prefix)
- Other languages at `/fr`, `/es`, `/it`, `/de`
- Browser language detected on first visit; preference persisted via `NEXT_LOCALE` cookie

---

## How it works

```
┌─────────────────┐     pnpm translate     ┌─────────────────────┐
│ messages/en.json│ ─────────────────────► │ messages/{fr,es,    │
│  (source)       │   google-translate-    │  it,de}.json        │
└─────────────────┘   api-x (free)         │  (auto-generated)   │
                                            └──────────┬──────────┘
                                                       │
                          next-intl reads at request time
                                                       ▼
                                       ┌────────────────────────────┐
                                       │ app/[locale]/layout.tsx    │
                                       │ - sets <html lang>         │
                                       │ - hreflang alternates      │
                                       │ - per-locale metadata      │
                                       └────────────┬───────────────┘
                                                    │
                                       ┌────────────▼───────────────┐
                                       │ proxy.ts (Next.js 16)      │
                                       │ - detects browser locale   │
                                       │ - redirects / → /fr etc.   │
                                       └────────────────────────────┘
```

**Stack:**
- [`next-intl`](https://next-intl.dev) `^4.11` — i18n framework, routing, message resolution
- [`google-translate-api-x`](https://www.npmjs.com/package/google-translate-api-x) — free unofficial Google Translate client (no API key)
- Next.js 16 `proxy.ts` (formerly `middleware.ts`) for locale detection

```

├── i18n/
│   ├── routing.ts          # Locale list, default, prefix strategy
│   ├── navigation.ts       # Locale-aware <Link>, useRouter, usePathname

├── messages/
│   ├── en.json             # ⭐ Source of truth — edit this

│   ├── es.json

│   ├── de.json
│   └── .translations.lock.json  # Tracks source fingerprints (commit this)
├── scripts/

├── components/
│   ├── language-switcher.tsx

└── app/[locale]/
    ├── layout.tsx          # <html lang>, metadata, NextIntlClientProvider
    └── page.tsx
```

## Day-to-day workflows

### 1. Change existing copy

Edit `messages/en.json`, then re-translate:

```bash
pnpm translate
```

The script only re-translates keys whose English source changed since the last run (tracked in `.translations.lock.json`). Untouched keys keep their existing translations — including any manual edits.

### 2. Add new copy in a component

**Server component:**
```tsx
import { getTranslations } from "next-intl/server";

export async function MySection() {
  const t = await getTranslations("MySection");
  return <h2>{t("title")}</h2>;
}
```

**Client component:**
```tsx
"use client";
import { useTranslations } from "next-intl";

export function MyButton() {
  const t = useTranslations("MyButton");
  return <button>{t("label")}</button>;
}
```

Then add the new key to `messages/en.json`:
```json
{
  "MySection": {
    "title": "Our story"
  },
  "MyButton": {
    "label": "Click me"
  }
}
```

Run `pnpm translate` to fan it out to the other languages.

### 3. Manually fix a single translation

Just edit the locale JSON directly (e.g., `messages/fr.json`). Your edit is preserved on subsequent `pnpm translate` runs as long as the English source for that key doesn't change.

> Why it works: the lock file stores the English string that *was* translated. If `en.json[key]` matches the lock entry, the script considers the translation up to date and skips it.

### 4. Force re-translate everything

```bash
pnpm translate --force
```

**Warning:** this overwrites all manual edits in non-English files.

### 5. Add a new language

1. Add the locale code to `i18n/routing.ts`:
   ```ts
   locales: ["en", "fr", "es", "it", "de", "pt"],   // added pt
   ```
2. Add the native label to the same file:
   ```ts
   export const localeLabels: Record<Locale, string> = {
     en: "English", fr: "Français", es: "Español",
     it: "Italiano", de: "Deutsch", pt: "Português",
   };
   ```
3. Add the locale to `TARGET_LOCALES` in `scripts/translate.mjs`.
4. Run `pnpm translate` — it will create `messages/pt.json`.

The new language appears in the language switcher and at `/pt` automatically.

### 6. Build for production

```bash
pnpm build
```

`prebuild` runs `pnpm translate` automatically, so any missing translations are generated before Next.js compiles. The build statically pre-renders all locales.

## Working with rich text (`<em>`, etc.)

Some strings include inline emphasis. Use `t.rich()`:

```tsx
// JSON
{ "title": "Building beyond the <em>known horizon.</em>" }

// Component
<h1>
  {t.rich("title", { em: (chunks) => <em>{chunks}</em> })}
</h1>
```

For translated **arrays of objects** (e.g. `Vision.principles`, `Projects.items`, `Services.items`), use `t.raw()` and render the HTML with `dangerouslySetInnerHTML` (the source is our own JSON, not user input):

```tsx
const items = t.raw("items") as Item[];
items.map((p) => <h3 dangerouslySetInnerHTML={{ __html: p.title }} />);
```

## Adding metadata (SEO)

Per-locale `<title>` and `<meta description>` come from `messages/{locale}.json` under the `Metadata` namespace. They're consumed by `app/[locale]/layout.tsx` via `generateMetadata`, which also emits `<link rel="alternate" hreflang="...">` for each locale and `x-default` for crawlers.

To change the canonical domain, update `baseUrl` in `app/[locale]/layout.tsx`.

## Gotchas

### Apostrophes before `<`, `{`, `}`, `#`

`next-intl` uses ICU MessageFormat. A bare `'` immediately followed by `<`, `{`, `}`, or `#` opens a quoted-literal section. Without a closing `'`, the rest of the string is treated as plain text and tags render as `&lt;em&gt;`.

**Wrong (renders `<em>` literally):**
```json
"titleLine2": "l'<em>horizon connu.</em>"
```

**Right (escape the apostrophe by doubling it):**
```json
"titleLine2": "l''<em>horizon connu.</em>"
```

`scripts/translate.mjs` does this escape automatically for new translations. If you hand-edit a string, remember the `''` rule when an apostrophe sits right before a tag.

### Brand names inside translatable strings

Tags like `Webflow`, `Next.js`, `iOS`, `SwiftUI` are translatable text from the JSON's perspective. Google usually preserves them, but occasionally a name gets translated (e.g., `Backend` → `Bagaglio`). To pin a value across re-translations:

1. Manually fix it in `messages/{locale}.json`.
2. Don't change the English source — the lock file will preserve your override.

### Translation quality

`google-translate-api-x` calls Google's free public endpoint. Quality is good for marketing copy, occasionally awkward for poetic/idiomatic phrases. For higher quality, swap the translation backend in `scripts/translate.mjs` for:

- **Google Cloud Translation API** (paid, requires API key, ~$20/M chars)
- **DeepL API** (paid, requires API key, ~30 European languages, often more natural)

The script's structure (read source → translate leaf strings → write JSON) stays the same; only the `translateString()` body needs to change.

### `localePrefix: "as-needed"`

The default locale (English) lives at `/`, not `/en`. This is set in `i18n/routing.ts`. If you ever need a hard `/en/...` prefix for English too (some SEO setups prefer this), change it to `"always"`.

## Reference: locale switcher

The dropdown is `components/language-switcher.tsx`. It:

- Reads available locales from `i18n/routing.ts`
- Shows native names (Deutsch, Français, …) — no flags (best practice — flags represent countries, not languages)
- Calls `router.replace(pathname, { locale: next })` on selection — `next-intl` rewrites the URL and sets the `NEXT_LOCALE` cookie
- Uses `aria-haspopup="listbox"` + Escape-to-close + click-outside-to-close

Styled in `app/globals.css` under `.lang`, `.lang-trigger`, `.lang-menu`.
