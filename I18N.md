# Internationalization (i18n)

The site ships in **English, French, Spanish, Italian, German**. Translations live as static JSON files (`messages/{locale}.json`), authored manually for accuracy and brand voice. Served via [`next-intl`](https://next-intl.dev) with full SSG, full SEO, and zero runtime translation cost.

- English at `/` (default locale, no prefix)
- Other languages at `/fr`, `/es`, `/it`, `/de`
- Browser language detected on first visit; preference persisted via the `NEXT_LOCALE` cookie

---

## How it works

```
messages/en.json  ◀── source of truth (English)
messages/fr.json  ◀── manually translated
messages/es.json
messages/it.json
messages/de.json
       │
       │  next-intl reads at request time
       ▼
┌────────────────────────────┐
│ app/[locale]/layout.tsx    │
│ - sets <html lang>         │
│ - hreflang alternates      │
│ - per-locale metadata      │
└────────────┬───────────────┘
             │
┌────────────▼───────────────┐
│ proxy.ts (Next.js 16)      │
│ - detects browser locale   │
│ - redirects / → /fr etc.   │
└────────────────────────────┘
```

**Stack:**
- [`next-intl`](https://next-intl.dev) `^4.11` — i18n framework, routing, ICU MessageFormat
- Next.js 16 `proxy.ts` (formerly `middleware.ts`) for locale detection
- No translation scripts, no third-party translation APIs

## Day-to-day workflows

### 1. Change existing copy

1. Edit `messages/en.json`.
2. Open the four other locale files (`fr.json`, `es.json`, `it.json`, `de.json`) and translate the same key in each.
3. Keep the JSON shape (keys, namespaces, array structure) **identical** across all locales — only the string values change.

### 2. Add new copy in a component

**Server component:**
```tsx
import { getTranslations } from "next-intl/server";

export async function MySection() {
  const t = await getTranslations("MySection");
  return <h2>{t("title")}</h2>;
}
```

**Client component:**
```tsx
"use client";
import { useTranslations } from "next-intl";

export function MyButton() {
  const t = useTranslations("MyButton");
  return <button>{t("label")}</button>;
}
```

Then add the key to **all five** `messages/*.json` files:
```json
{
  "MySection": { "title": "Our story" },
  "MyButton":  { "label": "Click me" }
}
```

If a key only exists in `en.json`, missing-key warnings will appear in the dev console for the other locales — and the English fallback may or may not render depending on the call site. Always keep the five files in sync.

### 3. Add a new language

1. Create `messages/{locale}.json` by copying `messages/en.json` and translating each value.
2. Add the locale code to `i18n/routing.ts`:
   ```ts
   locales: ["en", "fr", "es", "it", "de", "pt"],   // added pt
   ```
3. Add the native label to the same file:
   ```ts
   export const localeLabels: Record<Locale, string> = {
     en: "English", fr: "Français", es: "Español",
     it: "Italiano", de: "Deutsch", pt: "Português",
   };
   ```

The new language appears in the language switcher and at `/pt` automatically — no other code changes needed.

### 4. Build for production

```bash
pnpm build
```

The build statically pre-renders every locale via `generateStaticParams()` in `app/[locale]/layout.tsx`.

## Working with rich text (`<em>`, etc.)

Some strings include inline emphasis. Author the tag right inside the string and render with `t.rich()`:

```tsx
// JSON
{ "title": "Building beyond the <em>known horizon.</em>" }

// Component
<h1>
  {t.rich("title", { em: (chunks) => <em>{chunks}</em> })}
</h1>
```

For translated **arrays of objects** (e.g. `Vision.principles`, `Projects.items`, `Services.items`), use `t.raw()` and render the HTML with `dangerouslySetInnerHTML` — the source is our own JSON, not user input:

```tsx
const items = t.raw("items") as Item[];
items.map((p) => <h3 dangerouslySetInnerHTML={{ __html: p.title }} />);
```

## SEO metadata

Per-locale `<title>` and `<meta description>` come from each `messages/{locale}.json` under the `Metadata` namespace. They are consumed by `app/[locale]/layout.tsx` via `generateMetadata`, which also emits:

- `<link rel="alternate" hreflang="...">` for each locale
- `<link rel="alternate" hreflang="x-default" ...>` for crawlers
- a per-locale `<link rel="canonical">`

To change the canonical domain, edit `baseUrl` in `app/[locale]/layout.tsx`.

## Gotchas

### Apostrophes before `<`, `{`, `}`, `#` (ICU escape rule)

`next-intl` uses ICU MessageFormat. A bare `'` immediately followed by `<`, `{`, `}`, or `#` opens a quoted-literal section. Without a closing `'`, the rest of the string is treated as plain text — the tag will render as `&lt;em&gt;`.

This matters for languages with elision (French, Italian) when the article lands right before a tag.

**Wrong (renders `<em>` literally):**
```json
"titleLine2": "l'<em>horizon connu.</em>"
```

**Right (escape the apostrophe by doubling it):**
```json
"titleLine2": "l''<em>horizon connu.</em>"
```

In ICU, `''` always means one literal `'`. Apostrophes inside or *not* adjacent to `<`/`{`/`}`/`#` (e.g. `c'est`, `l'ennemi`, `n'est`, `un'occasione`) need **no** escaping.

Quick audit:
```bash
grep -nE "'[<{}#]" messages/*.json
# any matches need '' escaping
```

### Brand names and tech tags

Tags like `Webflow`, `Next.js`, `iOS`, `SwiftUI`, `React Native`, `Backend` — and project names like `Atlas`, `Northstar`, `Tessera` — are intentionally left untranslated across all locales. Project sub-names (`Field`, `OS`, `Audio`, `Health`, `Capital`, `Letters`) are part of the brand and stay in English too.

If you add a new project, follow the same convention.

### Number formatting

European locales (FR, IT, DE) use a comma as the decimal separator; ES (peninsular) and EN use a period. Coordinate strings in the locale files reflect this:
- `LAT 48.86° · LNG 2.34°` → EN, ES
- `LAT 48,86° · LNG 2,34°` → FR, IT, DE

When adding new numeric content, follow the convention of the locale.

### `localePrefix: "as-needed"`

The default locale (English) lives at `/`, not `/en`. This is set in `i18n/routing.ts`. If you ever need a hard `/en/...` prefix for English too (some SEO setups prefer this), change it to `"always"`.

## Reference: locale switcher

The dropdown is `components/language-switcher.tsx`. It:

- Reads available locales from `i18n/routing.ts`
- Shows native names (Deutsch, Français, …) — **no flags** (best practice — flags represent countries, not languages)
- Calls `router.replace(pathname, { locale: next })` on selection — `next-intl` rewrites the URL and sets the `NEXT_LOCALE` cookie
- Uses `aria-haspopup="listbox"` + Escape-to-close + click-outside-to-close

Styled in `app/globals.css` under `.lang`, `.lang-trigger`, `.lang-menu`.
